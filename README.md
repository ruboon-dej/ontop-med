# Ontop-MED

**Ontop Implementation with ELH Similarity Score**

A Virtual Knowledge Graph (VKG) framework for the medical domain that integrates a relational medical database with ontology-based concept similarity scoring and natural language explanation generation.

---

## Overview

Ontop-MED provides a REST API that accepts two medical concept names and returns:
- A structured SPARQL query result from the knowledge graph
- A semantic similarity score computed using the ELH Description Logic similarity algorithm (`sim_ε`)
- A human-readable explanation of why the two concepts are similar, generated by a local LLM

**Example request:**
```bash
curl "http://localhost:8090/api/v1/compare?concept1=Aspirin&concept2=Ibuprofen"
```

**Example response:**
```json
{
  "concept1": "Aspirin",
  "concept2": "Ibuprofen",
  "score": 0.66667,
  "backtrace": "{\"similarity\":0.66667,\"forward\":{...},\"backward\":{...}}",
  "explanation": "Aspirin and Ibuprofen share a high degree of similarity...",
  "sparqlResult": "{\"head\":{...},\"results\":{...}}"
}
```

## Architecture

The system is composed of five layers:

```
User / Client
      │  HTTP GET /api/v1/compare
      ▼
┌─────────────────────────────────────┐
│  Spring Boot API  (port 8090)       │  ← Layer 0: orchestration
│  MedVKGController.java              │
└───────────┬─────────────┬───────────┘
            │             │
     ┌──────┘    ┌────────┘         ┌──────────────┐
     ▼           ▼                  ▼              │
┌─────────┐ ┌──────────┐   ┌──────────────┐       │
│ Ontop   │ │  ELH     │   │  LLM         │       │
│ VKG     │ │  Explainer│   │  (Ollama)    │       │
│ :8080   │ │  layer   │   │  :11434      │       │
└────┬────┘ └──────────┘   └──────────────┘       │
     │                                             │
     ▼                                             │
┌─────────────────────────────────────┐            │
│  PostgreSQL  (port 5432)            │ ◄──────────┘
│  drugs, substances, compositions,   │
│  interactions tables                │
└─────────────────────────────────────┘
```

### Layer descriptions

| Layer | Component | Port | Role |
|-------|-----------|------|------|
| Spring Boot API | `MedVKGController.java` | 8090 | Entry point, orchestrates all layers |
| Ontop VKG | Docker container | 8080 | Translates SPARQL → SQL against PostgreSQL |
| ELH Explainer | `sim-elh-explainer.jar` | — | Computes DL ELH similarity score + backtrace |
| LLM | Ollama + llama3 | 11434 | Translates backtrace JSON → natural language |
| PostgreSQL | Docker container | 5432 | Stores raw medical data |

## Setup

### 1. Clone and configure environment

```bash
git clone <repository-url>
cd ontop-med
cp .env.example .env
# Edit .env with your database credentials and API keys
```

### 2. Download PostgreSQL JDBC driver

```bash
mkdir -p jdbc
wget -O jdbc/postgresql.jar https://jdbc.postgresql.org/download/postgresql-42.7.3.jar
```

### 3. Build the ELH similarity library

```bash
cd similarity/sim-elh-explainer-jar
mvn clean package -DskipTests
cp target/library-0.0.1.jar.original ../api/libs/sim-elh-explainer.jar
cd ../..
```

### 4. Start Docker stack

```bash
docker compose up -d
docker compose ps    # verify both containers are healthy
```

### 5. Load database schema and seed data

```bash
docker exec -i ontopmed-postgres psql -U ontopmed -d ontopmed < database/schema/01_schema.sql
docker exec -i ontopmed-postgres psql -U ontopmed -d ontopmed < data/seeds/seed_drugs.sql
```

### 6. Start Ollama

```bash
ollama serve    # run in a dedicated terminal
```

### 7. Start Spring Boot API

```bash
cd api
./gradlew bootRun
```

## Key Components

### MedVKGController.java

The main REST controller. Accepts two concept names, orchestrates calls to all three layers (Ontop, ELH, LLM), and returns a unified JSON response.

```
GET /api/v1/compare?concept1={name}&concept2={name}
```

### QueryService.java

Sends a SPARQL query to the Ontop endpoint at `http://localhost:8080/sparql`. Ontop rewrites the SPARQL into SQL using the OBDA mappings in `mapping/drugs.obda` and executes it against PostgreSQL.

### SimilarityBridge.java

Wraps the `sim-elh-explainer.jar` library. Instantiates `SimExplainer` with the ontology file at `ontology/custom-med.owl`, calls `similarity(DYNAMIC_SIM, concept1, concept2)`, and returns:
- `score` — a `BigDecimal` between 0 and 1
- `backtrace` — a JSON object describing which shared primitive concepts and roles contributed to the score

The similarity algorithm used is `DYNAMIC_SIM` (bottom-up dynamic programming with memoization), which is the most efficient method for larger ontologies. See Racharak (2021) for the formal definition of `sim_ε`.

### ExplanationConverterService.java

Takes the backtrace JSON from `SimilarityBridge` and sends it to the configured LLM provider along with the prompt template from `explanation/prompts/sim-elh-explainer-to-text.txt`. Returns a human-readable paragraph.

LLM provider is configured in `api/src/main/resources/application.properties`:

```properties
llm.provider=ollama    # local, free, private — default
llm.provider=openai    # requires OPENAI_API_KEY in .env
llm.provider=claude    # requires ANTHROPIC_API_KEY in .env
```

### OBDA Mapping (drugs.obda)

Defines how PostgreSQL tables map to ontology classes:

```
mappingId   drugs-to-MedicinalProduct
target      :drugs/{id} a idmp:MedicinalProduct ; idmp:hasName {drug_name}^^xsd:string .
source      SELECT id, drug_name FROM drugs

mappingId   substances-to-Substance
target      :substances/{id} a idmp:Substance ; idmp:hasName {substance_name}^^xsd:string .
source      SELECT id, substance_name FROM substances
```

## ELH Similarity Library

The similarity computation uses `sim-elh-explainer-jar` by Racharak (ReaLearn Research Group, Tohoku University). The library implements the `sim_ε` algorithm defined in:

> Racharak, T. (2021). *On Approximation of Concept Similarity Measure in Description Logic ELH With Pre-Trained Word Embedding*. IEEE Access.

### How similarity is computed

Given two concept names and an OWL ontology:

1. The library traverses the concept hierarchy to build description trees for each concept
2. It computes a homomorphism degree between the two trees
3. The score is averaged across forward (A→B) and backward (B→A) directions
4. The backtrace JSON records which primitive concepts and existential roles contributed

### Preference profiles (not yet configured)

The library supports pre-computed BioWordVec similarity scores for primitive concepts via preference profile files:

```
similarity/preference-profiles/primitive-concepts-similarity.txt
similarity/preference-profiles/primitive-concept-importance.txt
similarity/preference-profiles/role-importance.txt
similarity/preference-profiles/role-discount-factor.txt
```

Currently `SimilarityBridge.java` passes `null` for all preference profiles — the score is based purely on ontological structure. When BioWordVec scores are available, update the `SimExplainer` instantiation in `SimilarityBridge.java`:

```java
SimExplainer explainer = new SimExplainer(
    ONTOLOGY_PATH,
    "similarity/preference-profiles/primitive-concept-importance.txt",
    "similarity/preference-profiles/role-importance.txt",
    "similarity/preference-profiles/primitive-concepts-similarity.txt",
    null,
    null
);
```

```bash
cd ~/Documents/ontop-med

  tests/SmokeTest.java -d tests/

Expected output:
```
=== Smoke Test: sim-elh-explainer-jar ===
Similarity(ActivePlace, Mangrove) = 0.8333...

The test uses `place.krss` — a small test ontology from Example 1 of Racharak's ISWC 2025 paper. The exact expected score depends on the ontology and preference profile configuration.
